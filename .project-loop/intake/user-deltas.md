### Review Delta: Fairness, Cooldown, Architecture, Full Testing

ID источника: `S003`

Исходный ввод:

```text
Пятое: leaderboard по streak сам по себе нечестный, если у одного человека 1 пункт, а у другого 8. Для вашего маленького круга это терпимо, но в MVP я бы показывал сразу две метрики: текущий streak и completion rate за 7 дней. Тогда человек с одной лёгкой привычкой не выглядит абсолютным победителем только за счёт простоты.

Шестое: random triggers по целям лучше делать не совсем случайными 10%. Случайность в продукте часто ощущается как хаос. Лучше сделать “редкие мягкие вставки” с deterministic throttle: не чаще 1 раза в 3 дня на пользователя и только если у него есть активные цели. Внутри можно оставить pseudo-random, но с cooldown.

Главный риск не в продукте, а в реализации: текущий бот маленький и явный, но уже имеет callbacks, pending inputs, worker transitions, topic bindings, progress outbox и Telegram message editing. Если просто скопировать today-flow два раза, код начнёт быстро гнить. Это прямо видно по текущей архитектуре: internal/bot сейчас маршрутизирует setup, Today, reports и alert acknowledgements; internal/app/today отвечает за daily task rules; internal/app/progress — за progress outbox. Новые домены лучше делать отдельными пакетами, а не раздувать internal/bot/service.go.

есть еще несколько замечаний их учти тоже

докер запустил - запиши что если надо можешь запускать докер сам

сделай полное тестирование проверь что код покрыт адекватными тестами
```

Нормализация:
- [x] требование: leaderboard явно показывает current streak и 7-day completion rate; ranking не выглядит чистым streak-only.
- [x] требование: вставки про цели имеют детерминированное ограничение не чаще 1 раза в 3 дня на пользователя и работают только при активных целях.
- [x] требование: новые домены отделяются от раздутого `internal/bot/service.go`; worker делегирует routine/goals app packages.
- [x] решение: Docker локально доступен, можно запускать Docker-команды для тестирования.
- [x] валидация: выполнить PostgreSQL-backed `go test ./...`, `make lint`, `make test`, локальную миграцию.
